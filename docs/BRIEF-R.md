# Addendum brief — Study R: fan-out interventions (can teaching fix coverage, and does our own advice survive measurement?)

**Pre-registration, committed before any scored R run.** Study Q
found that fan-out instructions defeat every context strategy:
models emit legal patches that stop partway through the target set
(100% of oracle-arm failures were partial coverage). Study P found
that worked examples teach op-class behavior for ~900 tokens. Study
R crosses them, and adds the measurement our own guidance is
currently missing: we published "decompose fan-out into
single-target edits (the 87–100% case)" on the strength of
single-edit numbers, but task-level success under decomposition is
per-edit reliability **compounded** across 2–32 targets. That claim
gets measured here, not inferred.

## Corpus, baselines

`corpus/fanout.json` unchanged (45 tasks, seed 20260710, 2–32
targets). Baselines reused, never re-run: Study Q's Q-view, Q-full,
and Q-search cells.

## Conditions (all F patch dialect, shipped applier, id-free instructions)

Two interventions, each applied to both context bases (the model
inversion in Q means neither base is privileged), plus the
decomposition pipeline:

- **R-exV / R-exF** — a worked fan-out example appended to the
  system prompt of the Q-view / Q-full condition respectively. The
  example (pre-registered below) lives on Study P's example tree,
  whose `steps` block has exactly three text-atoms: instruction
  generated by the committed fan-out describer, reply covering ALL
  three targets. The tree is rendered in the arm's own format (a
  minimal container view for R-exV; the full JSON tree for R-exF).
  Unit tests prove the example patch applies and equals the fan-out
  applier's output.
- **R-ckV / R-ckF** — the checklist block (pre-registered below)
  appended to the same two bases. Same goal stated as instruction
  rather than demonstration — the contrast with the example arms is
  diagnostic (stopping-rule failure vs needing a demonstration).
- **R-decomp** — the published advice, executed literally. The
  application enumerates the target ids (the corpus's own committed
  target sets — exactly the deterministic query an app would run)
  and issues ONE single-target edit per target, sequentially, each
  as a fresh conversation: minimal view focused on that target,
  instruction from the corpus's single-edit describer (ids named —
  the app knows them), Study I's FT prompt, standard ≤3 correction
  rounds per subtask. A failed subtask leaves its node unchanged
  and the pipeline continues. One record per task; per-subtask
  outcomes recorded in it.

Pre-registered fan-out example (both renderings share this
instruction and reply; ids are Study P's `ex-` namespace, colliding
with no corpus tree):

```
Edit request: Set "maxLength" to 64 on every text-atom inside the block named "steps".

Reply:
[{"op": "set-attribute", "id": "ex-t1", "key": "maxLength", "value": 64}, {"op": "set-attribute", "id": "ex-t2", "key": "maxLength", "value": 64}, {"op": "set-attribute", "id": "ex-t3", "key": "maxLength", "value": 64}]
```

Pre-registered checklist block:

```
Coverage rules:
- When an edit request applies to every node matching a description, first enumerate the ids of ALL matching nodes you can see, then emit one operation per id. A patch that covers only some of the matching nodes is wrong.
```

## Grading

Identical to Study Q: `equalModuloNewIds` primary; the committed
coverage/collateral anatomy secondary; target-count bins (2–3, 4–6,
7+). R-decomp additionally reports per-subtask validity and the
compounding curve (task success vs target count), the number the
published advice depends on.

## Models and cells

`anthropic/claude-sonnet-4.5` and `google/gemini-3.5-flash`.
45 tasks × 5 conditions × 2 models = **450 runs** (R-decomp runs
2–32 model calls per task, all view-sized).

## Pre-registered hypotheses (held loosely)

- **R-H1 (teaching transfers to coverage):** the example arms beat
  their bases per model (paired McNemar, R-exV vs Q-view and R-exF
  vs Q-full), with the gains concentrated in partial-coverage
  failures. An arm **rescues fan-out** if it significantly beats its
  base AND is not significantly below that model's best Study Q arm
  (sonnet: Q-view 31/45; gemini: Q-full 36/45).
- **R-H2 (demonstration vs instruction):** if the checklist arms
  match the example arms, Q's failure was a stopping rule the model
  will follow when told; if the example arms win, coverage needs
  demonstration, mirroring P's content-teaches finding.
- **R-H3 (the decomposition audit, measured not predicted):**
  R-decomp's task-level success and its compounding shape decide
  whether the published guidance stands. Per-subtask validity is
  expected near 100% (set-attribute and remove on a named id, view
  in hand); whether that survives 32-fold compounding is exactly
  what we do not know. If R-decomp underperforms the best prompt
  intervention, the guidance in REPORT/barkup docs/blog gets
  corrected — the advice was ours, and it gets the same treatment
  as any other hypothesis.

## Decision rule

The best-performing approach per model becomes the documented
fan-out guidance, with numbers, in REPORT and the barkup docs (and
a blog correction if decomposition loses). If nothing reaches the
model's best Q baseline, fan-out stays an open boundary and is
published as such. Prompt interventions are evaluated exactly as
pre-registered here — no post-hoc tuning; a failed intervention is
a result, not a draft.

## Protocol

`maxOutputTokens: 60000`, temperature 0, 1 attempt + ≤3 correction
rounds per call (per subtask for R-decomp), resumable JSONL
(`results/raw/studyr-<model>.jsonl`), `cacheReadTokens` recorded,
mechanical-failure rules as in Study H. **Expected spend $25–45**
(two full-tree prompt arms dominate); abort past $75.

Honesty rules unchanged: this brief, the example constant with its
tests, the checklist block, and the decomposition runner are all
committed before the first scored call; results publish whatever
they show.
