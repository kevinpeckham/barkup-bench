## Follow-up-dropout failure counts: 110 vs 114 vs 116 vs 137

Three distinct populations have been quoted; all reconcile from raw:

1. 110 — ORIGINAL main run (v1), parity, C + D, haiku + gemini only:
   failures with detail.phase1Correct = true (phase 2 never applied).
   haiku 54 + gemini 56 = 110. This is REPORT.md's "110 audited
   phase-2 failures" (H4 section, written from the main-run records,
   which carry no transcripts). Across all four models the same
   population is 116 (sonnet 3 + gpt-5.4 3 more).

2. 114 — Study G Phase A dropout audit (scripts/dropout-audit.ts): a
   FRESH re-run of the same cells under the v1 protocol with
   transcript capture (parity, C + D, haiku + gemini,
   results/raw/dropout-audit-*.jsonl, 80 records per model).
   Phase-2 failures with phase 1 correct and no harness error:
   haiku 58 + gemini 56 = 114 — every one classified "no-tool-call"
   from the transcript (the model replied DONE without calling any
   tool). This is the population behind the blog draft's "We had 114
   of these", and it is the only transcript-verified count. The 110
   vs 114 gap is run-to-run variance between the original run and the
   audit re-run of the same protocol (54 vs 58 haiku phase-2
   failures; gemini 56 in both).

3. 137 — total "fail" lines in results/dropout-audit.log = ALL audit
   re-run failures: the 114 above plus 23 phase-1 failures (haiku 3 +
   gemini 20) where the initial insert itself was wrong — a tools-
   accuracy problem, not follow-up dropout, so the audit correctly
   excludes them from classification.

So: "114" is correct for the draft IF stated as the transcript-audited
count from the Study G Phase A re-run (v1 protocol, parity, C + D,
haiku + gemini, phase 1 correct). If the draft instead means "failing
cases in the original benchmark run", the number is 110 (haiku +
gemini) or 116 (all four models), and those transcripts were never
captured — only the audit re-run's were.
